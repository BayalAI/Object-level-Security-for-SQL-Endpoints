**Full Load**
<UL style="padding: 0px 0px 0px 32px;margin: 0px 0px 16px;color: rgba(0, 0, 0, 0.9);font-family: &amp;quot;font-size: 15px;font-style: normal;font-weight: 400;letter-spacing: normal;text-align: start;text-indent: 0px;text-transform: none;word-spacing: 0px;white-space: normal;background-color: rgb(255, 255, 255)"><LI style="list-style: inherit;margin-top: 0.5em;margin-bottom: 0.5em"><P style="margin: 0px 0px 16px">When setting up a new environment: You need to populate the system with all historical data before transitioning to incremental loads.</P></LI><LI style="list-style: inherit;margin-top: 4px;margin-bottom: 0.5em"><P style="margin: 0px 0px 16px">Moving data from an old system to a new system (e.g., on-premises to cloud) typically involves a full historical load.</P></LI><LI style="list-style: inherit;margin-top: 4px;margin-bottom: 0.5em"><P style="margin: 0px 0px 16px">When the entire dataset needs to be reloaded due to schema changes, business logic adjustments, or large-scale data corrections.</P></LI><LI style="list-style: inherit;margin-top: 4px;margin-bottom: 0.5em"><P style="margin: 0px 0px 16px">If there are concerns about data consistency, quality issues, or historical errors in the dataset, you may need to run a full load to ensure the target system is synchronized and accurate.</P></LI><LI style="list-style: inherit;margin-top: 4px;margin-bottom: 0.5em"><P style="margin: 0px 0px 16px">If you’re conducting performance optimizations, such as re-indexing tables or re-organizing partitions, you may need to perform a full load to ensure data is restructured optimally.</P></LI></UL>

**Watermark**
<UL style="padding: 0px 0px 0px 32px;margin: 0px 0px 16px;color: rgba(0, 0, 0, 0.9);font-family: &amp;quot;font-size: 15px;font-style: normal;font-weight: 400;letter-spacing: normal;text-align: start;text-indent: 0px;text-transform: none;word-spacing: 0px;white-space: normal;background-color: rgb(255, 255, 255)"><LI style="list-style: inherit;margin-top: 0.5em;margin-bottom: 0.5em">You want to load only new or updated records from the source into the delta table based on a timestamp, thus improving performance and reducing the amount of data processed.</LI><LI style="list-style: inherit;margin-top: 4px;margin-bottom: 0.5em">When you're working with large datasets where full loads would be inefficient.</LI><LI style="list-style: inherit;margin-top: 4px;margin-bottom: 0.5em">Have watermark column (like: - last_updated) in source data.</LI></UL>

**SQL Except**
<UL style="padding: 0px 0px 0px 32px;margin: 0px 0px 16px;color: rgba(0, 0, 0, 0.9);font-family: &amp;quot;font-size: 15px;font-style: normal;font-weight: 400;letter-spacing: normal;text-align: start;text-indent: 0px;text-transform: none;word-spacing: 0px;white-space: normal;background-color: rgb(255, 255, 255)"><LI style="list-style: inherit;margin-top: 0.5em;margin-bottom: 0.5em"><P style="margin: 0px 0px 16px">Small to Medium Data Loads: When loading structured data from relational sources (e.g., SQL databases, Data Warehouses) with smaller or moderately sized datasets.</P></LI><LI style="list-style: inherit;margin-top: 4px;margin-bottom: 0.5em"><P style="margin: 0px 0px 16px">Incremental Loading in SQL Environments: Ideal for finding differences between source and target tables and loading only new or changed data, especially in Synapse SQL or SQL-based Lakehouse environments.</P></LI><LI style="list-style: inherit;margin-top: 4px;margin-bottom: 0.5em"><P style="margin: 0px 0px 16px">Simplicity: Great for teams familiar with SQL who need a simple way to perform incremental loads.</P></LI></UL>

**Spark Except**
<UL style="padding: 0px 0px 0px 32px;margin: 0px 0px 16px;color: rgba(0, 0, 0, 0.9);font-family: &amp;quot;font-size: 15px;font-style: normal;font-weight: 400;letter-spacing: normal;text-align: start;text-indent: 0px;text-transform: none;word-spacing: 0px;white-space: normal;background-color: rgb(255, 255, 255)"><LI style="list-style: inherit;margin-top: 0.5em;margin-bottom: 0.5em"><P style="margin: 0px 0px 16px">Large-Scale Data Loads: When working with large datasets in distributed environments such as Data Lakes, Lakehouses, or big data platforms where scalability and performance are critical.</P></LI><LI style="list-style: inherit;margin-top: 4px;margin-bottom: 0.5em"><P style="margin: 0px 0px 16px">Parallelized Processing: Use in scenarios where Spark's distributed computing capabilities can be leveraged for faster data loads.</P></LI></UL>

**Change Data Feed**
<UL style="padding: 0px 0px 0px 32px;margin: 0px 0px 16px;color: rgba(0, 0, 0, 0.9);font-family: &amp;quot;font-size: 15px;font-style: normal;font-weight: 400;letter-spacing: normal;text-align: start;text-indent: 0px;text-transform: none;word-spacing: 0px;white-space: normal;background-color: rgb(255, 255, 255)"><LI style="list-style: inherit;margin-top: 0.5em;margin-bottom: 0.5em"><P style="margin: 0px 0px 16px">If there is any wrong data processed during incremental load and wanted to process correct data which required rollback to particular version.</P></LI><LI style="list-style: inherit;margin-top: 4px;margin-bottom: 0.5em"><P style="margin: 0px 0px 16px">Ideal for ETL pipelines that need to capture row-level updates (inserts, updates, deletes) and load them into downstream data stores or services.</P></LI><LI style="list-style: inherit;margin-top: 4px;margin-bottom: 0.5em"><P style="margin: 0px 0px 16px">Suitable when you want to minimize data reprocessing, as CDF avoids full table scans by delivering only the changes.</P></LI></UL>

**Delta Merge**
<UL style="padding: 0px 0px 0px 32px;margin: 0px 0px 16px;color: rgba(0, 0, 0, 0.9);font-family: &amp;quot;font-size: 15px;font-style: normal;font-weight: 400;letter-spacing: normal;text-align: start;text-indent: 0px;text-transform: none;word-spacing: 0px;white-space: normal;background-color: rgb(255, 255, 255)"><LI style="list-style: inherit;margin-top: 0.5em;margin-bottom: 0.5em">When loading data incrementally, and there are updates, inserts, or deletes (<STRONG>Applicable only if source data contains any column to identify which record(s) need(s) to be deleted</STRONG>).</LI><LI style="list-style: inherit;margin-top: 4px;margin-bottom: 0.5em">When source data changes over time (e.g., a customer’s address changes, order statuses are updated).</LI><LI style="list-style: inherit;margin-top: 4px;margin-bottom: 0.5em">When ensuring the latest version of data from different systems is reflected in your Delta table</LI></UL>

**SCD Type 2**
<UL style="padding: 0px 0px 0px 32px;margin: 0px 0px 16px;color: rgba(0, 0, 0, 0.9);font-family: &amp;quot;font-size: 15px;font-style: normal;font-weight: 400;letter-spacing: normal;text-align: start;text-indent: 0px;text-transform: none;word-spacing: 0px;white-space: normal;background-color: rgb(255, 255, 255)"><LI style="list-style: inherit;margin-top: 0.5em;margin-bottom: 0.5em"><P style="margin: 0px 0px 16px">Historical tracking is required: When you need to maintain historical versions of data records for auditing, analysis, or compliance reasons. For example, if you need to analyze how a customer's attributes (e.g., their address or preferences) have changed over time.</P></LI><LI style="list-style: inherit;margin-top: 4px;margin-bottom: 0.5em"><P style="margin: 0px 0px 16px">Non-volatile data changes: When changes in your data are infrequent but need to be stored and tracked in a versioned manner.</P></LI><LI style="list-style: inherit;margin-top: 4px;margin-bottom: 0.5em"><P style="margin: 0px 0px 16px">Data lineage and traceability: When you need to keep track of the "state" of a record at various points in time and identify which version of a record was valid at any given moment</P></LI></UL>
